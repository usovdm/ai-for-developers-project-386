# Спецификация приложения для бронирования встреч

## Краткое описание

Это простое приложение для бронирования встреч с одним владельцем календаря: владелец настраивает типы событий и доступность, а гости без регистрации выбирают доступный слот и создают бронирование.

## Роли

В проекте две условные роли:

- **Владелец календаря (админ)** — один заранее заданный профиль. Управляет типами событий, настройками доступности и предстоящими бронированиями.
- **Гость** — бронирует слоты без создания аккаунта и без входа в систему.

Регистрации и полноценной авторизации гостей нет.

## Доступ к разделам

- **Гостевая часть** — `/`, доступна всем.
- **Админская часть** — `/admin`, вход по форме логин/пароль. Учётные данные захардкожены в коде.

## Страницы

### Гостевая часть

Одна страница с календарём:

- над календарём — фильтр выбора типа события;
- детали слота, создание и удаление бронирования — в модальном окне.

Если типов событий ещё нет, вместо календаря показывается текстовое сообщение о том, что бронирование пока недоступно.

### Админская часть

Три страницы:

1. **Типы событий** — создание, редактирование, удаление.
2. **Настройки доступности** — рабочие дни и рабочее окно для всех типов событий.
3. **Предстоящие встречи** — список будущих бронирований всех типов.

## Сущности (продуктовый уровень)

### Тип события

Шаблон для будущих бронирований. Конкретная встреча появляется только после бронирования гостем.

**Поля, задаваемые админом:**

- название (обязательно);
- описание (обязательно);
- длительность: 15, 30 или 45 минут (обязательно).

**Назначается системой:**

- `id`;
- цвет из палитры 6 Google-цветов (случайно при создании, повторы допускаются).

Длительность одновременно определяет длительность встречи и шаг сетки слотов для этого типа.

Демо-типов по умолчанию нет.

**Правила:**

- админ может создавать, редактировать и удалять типы событий;
- удалить тип нельзя, если у него есть будущие бронирования;
- изменение типа события не меняет уже созданные бронирования; новые бронирования используют актуальные настройки типа.

### Настройки доступности

Единые для всех типов событий. Задаются админом на отдельной странице.

**Поля:**

- рабочие дни недели;
- время начала рабочего окна;
- время окончания рабочего окна.

**Ограничения:**

- время начала и окончания выбираются с шагом 15 минут;
- конец рабочего окна должен быть позже начала.

**Значения по умолчанию:**

- дни: Пн-Пт;
- окно: 09:00-18:00.

Изменение настроек доступности не влияет на уже созданные бронирования.

### Бронирование

Конкретная запись на выбранный временной слот.

**Поля, задаваемые гостем:**

- название бронирования (обязательно);
- имя (обязательно);
- email (обязательно);
- комментарий (необязательно).

## Календарь

### Общие правила

- Все даты и время считаются в часовом поясе `Europe/Moscow`.
- Доступное окно бронирования — ближайшие 14 дней, начиная с текущей даты.
- На одно и то же время нельзя создать две записи: проверяется пересечение интервалов `start-end`, а не только совпадение времени начала. Это правило действует для всех типов событий.
- При создании бронирования бэкенд повторно проверяет фактическую доступность слота. Если интервал уже занят, бронирование не создаётся; гость видит попап с ошибкой.

### Вид календаря

- Недельный календарь.
- Навигация стрелками на неделю назад/вперёд.
- Переход за пределы окна ближайших 14 дней невозможен; соответствующая стрелка `disabled`.
- Показываются только рабочие дни и рабочие часы из настроек доступности.

### Фильтр типа события

- Фильтр не выбран — календарь показывает все типы событий (режим обзора).
- В этом режиме видны все занятые слоты; бронирование недоступно; удаление занятого слота доступно.
- После выбора типа в фильтре гость может создавать бронирования для этого типа.

### Состояния слотов

- **Свободен** — доступен для бронирования при выбранном типе события.
- **Занят** — есть бронирование; можно открыть детали и попытаться удалить.

Время вне рабочего расписания и прошедшее время сегодняшнего дня отображаются серым фоном и недоступны для взаимодействия.

### Отображение занятых слотов

В занятом слоте гостю показываются:

- название бронирования;
- время;
- тип события.

Имя гостя, email и комментарий не показываются.

Типы событий визуально различаются цветом, назначенным типу при создании.

## Email

### Подтверждение создания бронирования

После успешного создания бронирования гость получает email с деталями: название бронирования, тип события, дата и время.

### Удаление бронирования

Удаление подтверждается кодом, отправленным на email.

Сценарий:

1. Гость нажимает на занятый слот в календаре.
2. Вводит email.
3. Если на этот слот есть бронирование с таким email, система отправляет код.
4. После ввода правильного кода бронирование удаляется.

Отдельного поиска «моих бронирований» нет. Перенос бронирований не поддерживается.

### Dev-режим

На первом этапе реальная отправка email через внешний провайдер не требуется. Письма доступны через локальные логи или dev-интерфейс.

Владелец календаря email-уведомления не получает.

## Админские возможности

### Типы событий

- создавать, редактировать, удалять;
- удаление запрещено при наличии будущих бронирований этого типа.

### Настройки доступности

- задавать рабочие дни и рабочее окно для всех типов событий.

### Предстоящие встречи

Список будущих бронирований всех типов, отсортированный по дате и времени по возрастанию.

В каждой записи:

- дата;
- время;
- тип события;
- название бронирования;
- имя гостя;
- email гостя;
- комментарий (если есть).

Админ может удалять любое бронирование без email-кода.

История прошедших бронирований в UI не отображается.

## Порядок реализации

1. Создание типов событий (админка).
2. Read-only календарь (можно использовать до появления бронирования).
3. Бронирование гостем.
4. Удаление бронирования гостем через email-код.
5. Остальные админские сценарии.

## Основные пользовательские потоки

### 1. Админ создаёт тип события

1. Входит в `/admin`.
2. Открывает страницу типов событий.
3. Создаёт тип: название, описание, длительность 15/30/45.
4. Система назначает `id` и случайный цвет.

### 2. Админ настраивает доступность

1. Открывает страницу настроек доступности.
2. Выбирает рабочие дни.
3. Задаёт начало и конец рабочего окна с шагом 15 минут.
4. Сохраняет настройки.

### 3. Гость создаёт бронирование

1. Открывает гостевой календарь.
2. Выбирает тип события в фильтре.
3. Нажимает на свободный слот.
4. В модальном окне заполняет: название бронирования, имя, email, комментарий (опционально).
5. Подтверждает создание.
6. Получает email-подтверждение (в dev-режиме — через логи/dev-интерфейс).

### 4. Гость удаляет бронирование

1. Нажимает на занятый слот в календаре.
2. Вводит email.
3. Получает код на email (если email совпадает с бронированием на этом слоте).
4. Вводит код.
5. Бронирование удаляется.

### 5. Админ управляет предстоящими встречами

1. Открывает страницу предстоящих встреч.
2. Просматривает список будущих бронирований.
3. При необходимости удаляет бронирование без email-кода.

## Не входит в MVP

- регистрация пользователей и полноценная авторизация гостей;
- перенос бронирований;
- поиск гостем своих бронирований;
- история прошедших бронирований в UI;
- интеграция с внешним календарём;
- реальные email-провайдеры;
- уведомления владельцу по email;
- индивидуальное расписание для каждого типа события;
- несколько владельцев календарей;
- таймзоны гостей.

## Критерии готовности MVP

- [ ] Админ может войти в `/admin` по захардкоженным логину/паролю.
- [ ] Админ может создать, отредактировать и удалить тип события.
- [ ] Админ может настроить рабочие дни и рабочее окно.
- [ ] Гость видит недельный календарь на ближайшие 14 дней.
- [ ] Гость может выбрать тип события и создать бронирование на свободный слот.
- [ ] Занятые интервалы блокируют пересекающиеся бронирования любых типов.
- [ ] Гость может инициировать удаление занятого слота через email и код подтверждения.
- [ ] Админ видит список предстоящих бронирований и может удалять их.
- [ ] Dev-email работает для подтверждения создания и кодов удаления.
