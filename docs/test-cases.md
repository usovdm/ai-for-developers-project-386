# Test cases

Документ фиксирует пользовательские и API-сценарии для будущей автоматизации. Источник требований: `docs/product.md`, `docs/product-spec.md`, `typespec/main.tsp`.

## Обозначения

- `P0` - критичный сценарий MVP.
- `P1` - важное бизнес-правило или пограничный сценарий.
- `P2` - дополнительная проверка контракта или UX-состояния.
- `API` - сценарий удобно покрывать на уровне HTTP API.
- `E2E` - сценарий нужно покрывать через пользовательский интерфейс.

## Admin auth

### TC-AUTH-001: Успешный вход админа

- Приоритет: `P0`
- Уровень: `API`, `E2E`
- Предусловия: известны захардкоженные логин и пароль владельца календаря.
- Шаги: открыть `/admin`; ввести корректные логин и пароль; подтвердить вход.
- Ожидаемый результат: админ получает доступ к админским разделам; API возвращает сессию с токеном.

### TC-AUTH-002: Отказ при неверных учетных данных

- Приоритет: `P0`
- Уровень: `API`, `E2E`
- Предусловия: нет.
- Шаги: открыть `/admin`; ввести неверный логин или пароль; подтвердить вход.
- Ожидаемый результат: доступ не предоставлен; API возвращает `unauthorized`.

### TC-AUTH-003: Защита админских endpoints

- Приоритет: `P0`
- Уровень: `API`
- Предусловия: нет валидного admin token.
- Шаги: выполнить запрос к любому admin endpoint без авторизации.
- Ожидаемый результат: запрос отклонен с `unauthorized`.

### TC-AUTH-004: Защита админских страниц от прямого открытия

- Приоритет: `P0`
- Уровень: `E2E`
- Предусловия: нет валидного admin token в браузере.
- Шаги: открыть напрямую `/admin/event-types`, `/admin/availability` или `/admin/bookings`.
- Ожидаемый результат: пользователь перенаправлен на `/admin`; показана форма входа; админские формы и списки не отображаются.

## Local integration

### TC-CORS-001: Backend разрешает dev origins frontend

- Приоритет: `P1`
- Уровень: `E2E`
- Предусловия: frontend и backend запущены в локальном dev/e2e окружении.
- Шаги: выполнить browser-like CORS preflight к backend с origins `http://127.0.0.1:<frontend-port>` и `http://localhost:<frontend-port>`.
- Ожидаемый результат: backend отвечает `200 OK` и возвращает соответствующий `access-control-allow-origin` для каждого origin.

## Recommended automation order

1. API smoke: `TC-AUTH-001`, `TC-EVENT-001`, `TC-AVAIL-001`, `TC-BOOK-001`, `TC-ADMIN-BOOK-001`.
2. Critical business rules: `TC-BOOK-005`-`TC-BOOK-009`, `TC-DEL-003`-`TC-DEL-005`, `TC-EVENT-009`.
3. E2E happy paths: admin creates event type, guest creates booking, guest deletes booking, admin deletes booking.
4. Calendar boundaries: `TC-CAL-001`-`TC-CAL-007`.
5. Contract and edge cases: `P1` and `P2` scenarios.
